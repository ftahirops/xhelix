Verification Engine

L0 hard-deny + L1 four-state lattice (VerifiedClean / BenignButUnverified / Suspicious / MaliciousLikely) using calibrated log-likelihood ratios across 8 independent domains.

The 8 verification domains

  1. Identity — binary hash, package signature, expected uid/gid, cgroup unit, capabilities
  2. Authority — role contract, credbroker grants, declared capability set
  3. Provenance — proctree lineage, SourceAnchor + CausalSet, taint inheritance
  4. Asset — asset class, service-role↔asset matrix, tenant boundary
  5. Behavioral — BRP envelope check + per-host overlay
  6. Temporal — maintenance window, deploy window, time-since-bootstrap, phase
  7. Cross-signal — eBPF ∩ FIM ∩ connstate ∩ memory ∩ identity agreement
  8. Threat — IOC match, honey-artifact touch, recent alert proximity, recent exploit proximity

Why LLR not naive Bayes

Naive Bayes assumes independent features. Our domains share underlying signals (eBPF + proctree + baseline all share the process stream). We use calibrated log-likelihood ratios with capped per-domain evidence + explicit interaction bonuses — the math from the v2 verification architecture doc.
