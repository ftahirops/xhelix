Three-plane architecture

The three planes are intentionally non-trusting of each other. Endpoint decides fast and locally. Hub understands deeply. Curator alone is allowed to change trust.

Why three planes and not one

Each plane represents a different kind of bet. If any one bet fails, the others must continue to provide value — particularly the endpoint, which must never depend on the hub being alive.

PlaneThe betIf it fails
Endpoint "I can decide in <2 seconds with what I see locally" Server is unprotected — unacceptable
Hub "I can find cross-host patterns nobody on one server can see" We lose analytical depth, but endpoint still protects
Curator "I can decide what's trustworthy enough to push to every server" Bad policy hits everyone — so we make this step the hardest, slowest, and most reviewed
Key safety property
The endpoint NEVER depends on the hub being alive. Hub down for 30+ days → server is still protected by what was last shipped to it. This rule was non-negotiable in the locked scope (FINAL_SCOPE_V2 §3.3).

Endpoint plane

One per protected host. Autonomous decision-maker. Owns:

  • L0 hard-deny enforcement — immutable invariants (protected-paths backstop, forever-suspicious actions)
  • L1 fast verification — BRP envelope check against cached signed profile
  • Source lineage — SourceAnchor mint + CausalSet propagation
  • Immediate containment — kill / block / quarantine / freeze, <2s decision→enforce
  • Forensic chain signing — Ed25519 batches, offline-verifiable
  • Telemetry classification — Class A/B/C upload to hub (if reachable)

What the endpoint cannot do:

  • Pull unsigned policy from anywhere
  • Self-modify hard invariants
  • Depend on hub for enforcement decisions

Hub plane

One per fleet (typically; regional federation deferred to post-v2). Analysis only.

  • Telemetry ingestion via mTLS
  • Time-series + graph storage (TimescaleDB on Postgres — see Data Flow)
  • Cross-host source graph reconstruction
  • L2 deep multi-chain composition
  • Per-source aggregate scoring at fleet scale
  • Analysis query API (the AI/observability surface)
  • Candidate BRP profile proposal — recommend only
The hub CANNOT publish policy
Hub being compromised must not let an attacker push malicious profiles to endpoints. The hub can recommend candidate profiles; it can never sign them. That privilege belongs exclusively to the curator.

Curator plane

Separate machine, typically air-gappable. The only plane allowed to change endpoint trust.

  • Candidate BRP review queue (input from hub)
  • Diff-against-prior-version explainer
  • Two-person approval workflow (cryptographic — each reviewer signs)
  • HSM-backed signing (YubiHSM 2 / AWS KMS / Vault transit)
  • Staged ring rollout (ring 0 → ring 1 → ring 2 → general)
  • Auto-rollback on detection rate or FP rate spike
  • Signed distribution channel to endpoints

Plane invariants (cannot be violated)

PlaneOwnsForbidden
Endpoint hard-deny, BRP match, source anchors, taint, immediate containment, cached signed policy, offline survival Cannot pull unsigned policy. Cannot self-modify hard invariants. Must work 30+ days without hub.
Hub telemetry ingestion, cross-host graph, deep analysis, candidate proposal Cannot publish policy directly. Cannot bypass curator. Compromise of hub does not compromise endpoints.
Curator candidate review, signing, staged rollout, rollback Cannot observe live telemetry. Cannot run analysis. Cannot push without two-person approval and HSM signature.
