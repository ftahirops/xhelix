2.5 Architecture Diagrams

Six diagrams showing how data and decisions flow through the system. Each one captures a different layer of the same architecture.

1. The three planes — what each owns, what flows between them

Three independent planes. Each box can only do what its role allows. Compromise of one plane cannot subvert another.

2. What happens to every kernel event on the endpoint

Every event flows through the same 3-gate pipeline. 99% are cleared silently at L2; 1% compose into incidents at L1; rare hard-deny matches kill the chain at L0 immediately.

3. Source attribution — one SSH login's full activity tree

One authenticated entry point (the SSH login) anchors EVERY child action. The anchor survives PID reuse, sudo escalation, and daemon restarts.

4. Three correlation levels — cost vs coverage

Three levels run in parallel. 99% of events stop at L1 (cleared). ~1% enter L2 attribution. The tiny fraction L2 flags as Suspicious get pulled into L3 deep composition on the hub.

5. Multi-chain convergence — how an incident emerges

An attacker can hide one suspicious step. They cannot hide three independent telemetry chains all pointing at the same authenticated session.

6. Data lifecycle — from kernel event to long-term archive

Each stage drops volume by 10-100×. By the time data reaches long-term storage it's compressed, deduped, and aggregated. The kernel sees billions of events; the 1-year archive holds thousands.
